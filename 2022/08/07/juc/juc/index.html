java 并发编程知识总结（一）

发表于 2022-08-07 更新于 2022-08-09 阅读次数：
出去面试的时候，总是有面试官喜欢问高并发，今天就在这里做一个详细的整理。

多线程的基本原理

  • 多线程

    • 线程从创建到最终执行的整体流程

      • 原理图
      • 线程启动后，并不会立即执行，而是要等到操作系统层面的CPU调度算法，把当前线程分配给某个CPU来执行。
      • 线程的运行状态，在java中一共有6种状态：
        • NEW，新建状态，也就是通过new Thread()时的状态。
        • RUNNABLE，运行状态，通过start方法启动线程后的状态。
        • BLOCKED，阻塞状态，当线程执行synchronized代码，并且未抢占到同步锁，会变成该状态。
        • WAITING，调用Object.wait()等方法，会让线程变成该状态。
        • TIMED_WAITING，超时等待状态，如sleep，超时候会自动唤醒。
        • TERMINATED，终止状态，线程的run方法中的指令执行完成后的状态。

    • 多线程的状态切换

      如上图所示意，当一个线程创建后，通过Thread.start方法后，会进入到READY状态，等待CPU调度，该线程会切换到RUNNING状态。

      • 当线程调用Object.wait()或者Thread.join以及LockSupport.park方法，该线程会进入到WAITING状态。
      • 当线程调用Object.notify/notifyAll(),以及LockSupport.unpark等方法，就会进入到RUNNABLE状态。
      • 当线程调用Thread.sleep,Object.wait,Thread.join以及LockSupport.parkNanos,LockSupport.parkUntil,Thread.sleep等方法会金瑞到TIMED_WAITING状态。
      • 当线程调用Object.notify/notifyAll(),以及LockSupport.unpark等方法，就会进入到RUNNABLE状态。
      • 当线程抢占到synchronized锁未成功时，会进入到blocked状态。当前抢占成功的时候，会进入到runnable状态。

    • java线程状态只有RUNNABLE状态，没有显示的READY和RUNNING区分，只有jvm系统级别实现了这种区分。

    • Thread.yield和Thread.sleep的区别，如上图所示意，Thread.yield由CPU调度算法调度，只不过是由Ready状态进入到了RUNNING状态。而Sleep则是由RUNNABLE进入到TIMED_WAITING状态。

  • 上下文切换带来的性能开销
    导致线程上下文切换的原因总结如下：

    • 多个任务抢占synchronized同步锁资源。
    • 在线程运行中存在这IO资源，CPU调度器会切换CPU时间片。
    • 在线程中通过主动阻塞当前线程的方法释放CPU时间片。
    • 当前线程执行完成后释放CPU时间片，CPU重新调度。

  • 理解上下文切换

    • 进程上下文切换
      • CPU时间片分配。
      • 当前进程系统资源（如内存）不足时，进程会被刮起。
      • 当存在优先级更高的进程运行时，当前进程有可能会被刮起，CPU时间片分配给优先级更高的进程运行。
        与线程上下文切换相比，进程上下文切换性能开销巨大。每次开销只有几十纳秒，但是频繁开销，CPU真正时间很少。进程每次切换都需要在操作系统层面保存用户空间的虚拟内存、栈、全局变量等状态，还要保存内核空间的内核堆栈、寄存器等状态。
    • 线程上下文切换
      • 当两个线程属于不同进程的同时，由于进程资源不足，所以线程的切换就是进程的切换。
      • 当两个线程属于同一个进程的时候，只需要保存线程的上下文。
        线程的上下文切换，需要保存上一个线程的私有数据、寄存器等数据，这个过程同样占用CPU资源，当上下文切换过于频繁时，会使得CPU不断进行切换，无法真正计算，最终导致性能下降。
    • 中断上下文切换
      • CPU本身故障、程序故障。
      • IO中断。

  • 如何减少切换上下文切换

    • 减少线程数
    • 采用无锁设计
    • 采用CAS做自选操作

深度理解synchronized实现原理

  • 导致线程不安全的原因主要有
    • 原子性
      • 产生原子性的原因有两个
        • CPU时间片切换
        • 执行指令的原子性，也就是线程运行的程序或者指令是否具备原子性。
    • 有序性
    • 可见性
      如果一个线程对一个共享变量进行了修改，而其他线程不能及时读取修改之后的值，那么我们可以认为多线程环境下存在变量可见性问题。

Java中的synchronized同步锁

  • synchronized同步锁的范围
    • 类锁
      类锁是全局锁，范围比较大，可以修饰静态方法，也可以修饰代码块。
    • 对象锁
      对象锁是实例锁，可修饰普通方法，也可以修饰代码块。
    • 同步锁的核心特征是排他。
  • 实现互斥锁满足两个条件
    • 必须竞争同一个共享资源。
    • 需要有一个标记来识别当前的锁的状态是空闲还是繁忙。

MarkWord

如上所示意，对象头中通过锁标记位来标记出偏向锁、锁标记位。可以看到，锁状态分为无锁、偏向锁、轻量级锁、重量级锁。

  • 锁的类型
    • 偏向锁
      在没有多线程的竞争的情况下访问synchronized修饰的代码块加锁场景。加锁是为了防范线程安全性的风险，在没有线程竞争的情况下去访问synchronized同步代码块，会尝试通过偏向锁来抢占防蚊子哥。如果抢占成功，则直接修改对象头中的锁标记。偏向锁为01，标记为01，以及线程threadId。偏向的意思是线程X获得了偏向锁，那么当线程X后续访问这个同步方法的时候，只需要判断对象头中的线程ID和线程X是否相等。如果相等，则不需要再次去抢占该锁，直接获得访问资格即可。
    • 轻量级锁
      • 原理如下
        • 偏向锁和轻量级锁的对比
          • 偏向锁，就是在一段时间之内只由同一个线程来获得和释放锁，加锁的方式是把ThreadId保存到锁对象的MarkWord中。
          • 轻量级锁，存在锁交替竞争的场景，在同一时刻不会有多个线程同时获得锁，它的实现方式在每个线程的栈帧中分配一个BasicObjectLock对象，然后把锁对象中的MarkWord
            拷贝到LockRecord中，最后把锁对象的MarkWord指针指向LockRecord。轻量级锁之所以这样设计，是因为锁对象在竞争的过程中可能会发生变化，但是每个线程的LockRecord的MarkWord不会受到影响。因此当触发锁膨胀，能够通过LockRecord和锁对象的MarkWOrd进行比较来判定在持有的轻量级锁的过程中，锁对象是否被其他线程抢占国，如果有，则需要在轻量级锁释放锁的过程中唤醒被阻塞的请他线程。
    • 重量级锁
      如果线程在运行synchronized 同步代码块时，发现锁状态是轻量级锁并且有其他线程抢占了锁资源，那么该线程就会触发锁膨胀升级到重量级锁。在获取重量级锁之前，会实现锁膨胀，在锁膨胀的方法中创建一个ObjectMonitor对象，然后把ObjectMonitor对象的指针保存在锁对象的MarkWord中。锁膨胀分为四种情况：
      • 当前已经是重量级锁的状态，不需要再膨胀，直接从锁对象的MarkWord中获取ObjectMonitor对象的指针返回即可。
      • 如果有其他线程正在进行锁膨胀，那么通过自旋的方式不断重试知道其他线程完成锁膨胀，其实就是创建一个ObjectMonitor对象。
      • 如果当前有其他线程获得了轻量级锁，那么当前线程会完成锁的膨胀。
      • 如果当前是无锁状态，也就是说之前所获得的锁的资源的线程正好释放了锁，那么当前线程需要完成锁膨胀。

死锁产生的必要条件

导致死锁的条件有四个：

  • 互斥条件，共享资源X和Y只能被一个线程占用。
  • 请求和保持条件，线程T1已经取得共享资源X，在等待共享资源Y的时候，不释放X。
  • 不可抢占条件，其他线程不能强行抢占线程T1的占有的资源。
  • 循环等待条件，线程T1等待线程T2占有的资源，线程T2等待线程T1占有的资源，即循环等。
    除了互斥条件不可破坏，其他都可以破坏。

volatile关键字

JVM volatile 修饰符主要用于以下
1，多线程场景当中，需要保证共享数据内存可见性的场景。
2，需要避免指令重排的场景

通常认为是不能达到保证原子性的目的，保证内存可见性，并且避免指令重排的场景。

深度理解可见性问题的本质

CPU缓存一致性问题

  • MESI 协议
    MESI中每个缓存行都有四个状态，分别是E（exclusive）、M（modified）、S（shared）、I（invalid）。下面我们介绍一下这四个状态分别代表什么意思。

    • M：代表该缓存行中的内容被修改了，并且该缓存行只被缓存在该CPU中。这个状态的缓存行中的数据和内存中的不一样，在未来的某个时刻它会被写入到内存中（当其他CPU要读取该缓存行的内容时。或者其他CPU要修改该缓存对应的内存中的内容时（个人理解CPU要修改该内存时先要读取到缓存中再进行修改），这样的话和读取缓存中的内容其实是一个道理）。

    • E：E代表该缓存行对应内存中的内容只被该CPU缓存，其他CPU没有缓存该缓存对应内存行中的内容。这个状态的缓存行中的内容和内存中的内容一致。该缓存可以在任何其他CPU读取该缓存对应内存中的内容时变成S状态。或者本地处理器写该缓存就会变成M状态。

    • S:该状态意味着数据不止存在本地CPU缓存中，还存在别的CPU的缓存中。这个状态的数据和内存中的数据是一致的。当有一个CPU修改该缓存行对应的内存的内容时会使该缓存行变成 I 状态。

    • I：代表该缓存行中的内容时无效的。

  • 总线锁和缓存锁机制

    • 总线 data bus 。CPU和内存，IO设备的公共通道。总线锁在总线上声明一个Lock#信号，这个信号能确保当前CPU可以访问，其他处理器会被阻塞，这就使得同一时刻只有一个处理器能访问共享内存，从而解决了缓存不一致的问题。但是这样会有性能问题。
    • 缓存锁，如果当前CPU访问的数据已经缓存在其他的CPU缓存中，那么CPU不会在总线上声明Lock#信号，而是采用MESI协议实现多个CPU的缓存一致性。

指令排序

  • 指令排序
    编译器或CPU为了优化程序的执行性能而对指令进行重新排序的一个手段，重排序会带来可见行问题。
    • 编译器重排序
    • CPU重排序。
  • 总结指令排序

JVM内存屏障指令

如本章上文所提到，通过volatile关键字来避免内存重排序。

  • LoadMemoryBarrier 该内存屏障主要用于保证高速缓存行在进行本地读操作时的内存可见性，即保证屏障后的本地读操作结果一定是最新的数据结果。

  • StoreMemoryBarrier 该内存屏障主要用于保证高速缓存写数据的内存可见性

  • FullBarrier 该内存屏障的性能不高，但是可以保证数据一致性，是操作系统提供的通用屏障。

  • LoadStoreBarrier 和loadMemoryBarrier没有区别。

// Implementation of class OrderAccess.

inline void OrderAccess::loadload()   { acquire(); }
inline void OrderAccess::storestore() { release(); }
inline void OrderAccess::loadstore()  { acquire(); }
inline void OrderAccess::storeload()  { fence(); }

inline void OrderAccess::acquire() {
  volatile intptr_t local_dummy;
#ifdef AMD64
  __asm__ volatile ("movq 0(%%rsp), %0" : "=r" (local_dummy) : : "memory");
#else
  __asm__ volatile ("movl 0(%%esp),%0" : "=r" (local_dummy) : : "memory");
#endif // AMD64
}

inline void OrderAccess::release() {
  // Avoid hitting the same cache-line from
  // different threads.
  volatile jint local_dummy = 0;
}

inline void OrderAccess::fence() {
  if (os::is_MP()) {
    // always use locked addl since mfence is sometimes expensive
#ifdef AMD64
    __asm__ volatile ("lock; addl $0,0(%%rsp)" : : : "cc", "memory");
#else
    __asm__ volatile ("lock; addl $0,0(%%esp)" : : : "cc", "memory");
#endif
  }
}

inline void     OrderAccess::store_fence(jbyte*  p, jbyte  v) {
  __asm__ volatile (  "xchgb (%2),%0"
                    : "=q" (v)
                    : "0" (v), "r" (p)
                    : "memory");
}

inline void     OrderAccess::store_fence(jlong*   p, jlong   v) {

      • 两个作用，一是解决缓存一致性问题。二是实现内存屏障。

volatile关键字代码总结

会调用OrderAccess:storeload()内存屏障方法，会基于lock指令来实现。

  • volatile关键字会在jvm层面声明一个C++ volatile，它能够防止JIT层面的指令重排。
    在修饰volatile关键字后，jvm会调用OrderAccess::storeload方法完成内存屏障，该方法定义了lock指令：
    • CPU层面，给volatile赋值的指令会先存储到storebuffer中，所以lock指令会使得storebuffer刷新到缓存行。
    • 其他CPU缓存行该volatile修饰的关键字值失效。
    • 其他线程再来获取volatile修饰关键字的值，会从内存中获取，使得线程能够获得volatile关键字修饰的最新值。

Wait 方法实现：

ObjectSynchronizer::wait方法

通过object的对象中找到ObjectMonitor对象

调用方法

void ObjectMonitor::wait(jlong millis, bool interruptible, TRAPS)

通过ObjectMonitor::AddWaiter调用把新建立的ObjectWaiter对象放入到 _WaitSet 的队列的末尾中

然后在ObjectMonitor::exit释放锁，接着 thread_ParkEvent->park 也就是wait

Notify方法的实现：

ObjectSynchronizer::notify方法 ，调用ObjectSynchronizer::inflate ，object的对象中找到ObjectMonitor对象 ，然后调用方法ObjectMonitor::notify ，调用ObjectMonitor::DequeueWaiter 摘除第一个ObjectWaiter对象从_WaitSet 的队列中
并把这个ObjectWaiter对象放入_EntryList中,_EntryList 存放的是ObjectWaiter的对象列表，列表的大小就是那些所有在等待这个对象锁的线程数。

注意这里并没有调用ObjectMonitor::exit释放锁

NotifyALL和Notify 的区别就是

通过遍历调用ObjectMonitor::DequeueWaiter，把所有的_WaitSet的队列中的ObjectWaiter对象放入到_EntryList中

关于放入到_EntryList的策略大概有4中Policy，其中还涉及到一个_cxq的队列，先不具体介绍了

notify, 和notifyAll 都没有释放对象的锁，而是在Synchronizer同步块结束的时候释放
