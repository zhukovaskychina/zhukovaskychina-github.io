java 并发编程知识总结（二）

发表于 2022-08-07 更新于 2022-08-09 阅读次数：
出去面试的时候，总是有面试官喜欢问高并发，今天就在这里做一个详细的整理。

条件等待机制

实现线程之间的通信方法如下：

  • 基于volatile修饰的共享变量。
  • 通过wait/notify机制。
  • Thread.join方法。
  • 使用Synchronized方法。
  • Condition.await/signal等方法。

wait/notify

当线程调用Object.wait()，该线程会进入到WAITING状态。当线程调用notify()方法，唤醒处于阻塞状态下的一个线程。
当线程调用Object.notifyAll(),唤醒处于阻塞状态下的所有线程。
其核心本质是针对同一个共享对象的竞争来实现数据变更的通知，也就是当某个共享变量满足某种条件时会触发阻塞和唤醒，从而实现线程的通信。

/**
 * 生产者和消费者，wait()和notify()的实现
 * @author ZGJ
 * @date 2017年6月22日
 */
public class Test1 {
    private static Integer count = 0;
    private static final Integer FULL = 10;
    private static String LOCK = "lock";
    
    public static void main(String[] args) {
        Test1 test1 = new Test1();
        new Thread(test1.new Producer()).start();
        new Thread(test1.new Consumer()).start();
        new Thread(test1.new Producer()).start();
        new Thread(test1.new Consumer()).start();
        new Thread(test1.new Producer()).start();
        new Thread(test1.new Consumer()).start();
        new Thread(test1.new Producer()).start();
        new Thread(test1.new Consumer()).start();
    }
    class Producer implements Runnable {
        @Override
        public void run() {
            for (int i = 0; i < 10; i++) {
                try {
                    Thread.sleep(3000);
                } catch (Exception e) {
                    e.printStackTrace();
                }
                synchronized (LOCK) {
                    while (count == FULL) {
                        try {
                            LOCK.wait();
                        } catch (Exception e) {
                            e.printStackTrace();
                        }
                    }
                    count++;
                    System.out.println(Thread.currentThread().getName() + "生产者生产，目前总共有" + count);
                    LOCK.notifyAll();
                }
            }
        }
    }
    class Consumer implements Runnable {
        @Override
        public void run() {
            for (int i = 0; i < 10; i++) {
                try {
                    Thread.sleep(3000);
                } catch (InterruptedException e) {
                    e.printStackTrace();
                }
                synchronized (LOCK) {
                    while (count == 0) {
                        try {
                            LOCK.wait();
                        } catch (Exception e) {
                        }
                    }
                    count--;
                    System.out.println(Thread.currentThread().getName() + "消费者消费，目前总共有" + count);
                    LOCK.notifyAll();
                }
            }
        }
    }
}

生产者消费者模式总结

wait/notify为何需要加同步锁

  • 其核心本质是针对同一个共享对象的竞争来实现数据变更的通知，为了保证原子性，需要加锁。
  • wait/notify方法需要实现线程的阻塞和唤醒，当某个线程调用notify唤醒时，唤醒哪个线程，从哪里去找？而synchronized提供了同步队列的机制，正好为wait/notify
    方法提供很好的协同机制。
  • 之所以放在Object中，是因为java任何一个Object都关联了一个监视器对象，能够很好的实现同步锁机制。

wait(0)的含义


导致当前线程等待，直到另一个线程调用该对象的notify()方法或notifyAll()方法，或者等待一段指定的时间。
当前线程必须拥有这个对象的监视器。
该方法导致当前线程(称为T)将自己放入该对象的等待集，然后放弃对该对象的任何和所有同步请求。为了线程调度的目的，线程T被禁用，并处于休眠状态，直到发生以下四种情况之一:
其他一些线程调用这个对象的notify方法，线程T碰巧被任意选择为要唤醒的线程。
其他一些线程调用这个对象的notifyAll方法。
其他一些线程中断线程T。
指定的实时时间已经过，或多或少。但是，如果超时为零，则不考虑实时时间，线程只是等待，直到通知。
然后，线程T从该对象的等待集中删除，并重新启用线程调度。然后，它以通常的方式与其他线程竞争在该对象上同步的权利;一旦它获得了对对象的控制，它对对象的所有同步声明将恢复到以前的状态—也就是说，恢复到调用wait方法时的状态。然后线程T从等待方法的调用中返回。因此，从wait方法返回时，对象和线程T的同步状态与调用wait方法时完全相同。
线程也可以在没有通知、中断或超时的情况下唤醒，即所谓的虚假唤醒。虽然这种情况在实践中很少发生，但应用程序必须通过测试应该导致线程被唤醒的条件来防范它，并在条件不满足时继续等待。换句话说，等待应该总是出现在循环中，就像下面这个:
同步(obj) {
While (&lt;条件不成立&gt;)
obj.wait(超时);
．.． //执行适合条件的操作
｝

关于这个主题的更多信息，请参阅Doug Lea的“Java并发编程(第二版)”中的第3.2.3节。(Addison-Wesley, 2000)，或Joshua Bloch的“有效的Java编程语言指南”(Addison-Wesley, 2001)中的第50项。
如果当前线程在等待之前或等待期间被任何线程中断，则会抛出InterruptedException异常。直到如上所述恢复该对象的锁状态时，才会引发此异常。
注意，wait方法将当前线程放入该对象的等待集，只解锁该对象;当线程等待时，当前线程可能被同步的任何其他对象保持锁定。
此方法只能由该对象监视器所有者的线程调用。有关线程成为监视器所有者的方法的说明，请参阅notify方法。

如上是机器翻译。
wait的标准范式如下：

synchronized (obj) {
    while (<condition does not hold>)
        obj.wait(timeout);
        ...  // Perform action appropriate to condition
}

If timeout is zero, however, then real time is not taken into consideration and the thread simply waits until notified.
如果超时为0，则real time不会被考虑，并且线程会等待一直到被通知。

注意，wait方法将当前线程放入该对象的等待集，只解锁该对象;当线程等待时，当前线程可能被同步的任何其他对象保持锁定。
此方法只能由该对象监视器所有者的线程调用。有关线程成为监视器所有者的方法的说明，请参阅notify方法。

Thread.join

Thread.join 核心实质是isAlive()和wait()。

public final native boolean isAlive();

public final native void wait(long timeout) throws InterruptedException;

public final synchronized void join(long millis) throws InterruptedException {
    long base = System.currentTimeMillis();
    long now = 0;
    if (millis < 0) {
        throw new IllegalArgumentException("timeout value is negative");
    }
    if (millis == 0) {
        while (isAlive()) {
            wait(0);
        }
    } else {
        while (isAlive()) {
            long delay = millis - now;
            if (delay <= 0) {
                break;
            }
            wait(delay);
            now = System.currentTimeMillis() - base;
        }
    }
}

Condition

Condition有两个方法，说明如下：

  • await()方法，让线程等待，并且释放锁。
  • signal/signalAll方法，唤醒await()方法阻塞的线程。

Condition和wait()/notify()的区别，
Condition可以针对不同的场景设置多个不同的Condition，当Condition.signal()方法被调用的时候，不需要唤醒所有的线程，只需要唤醒指定的Condition的线程即可，这样做可以减少线程的无效竞争。
